# UltraGesture 工作节点记录

## 1. 项目概述

**课题**: 基于FMCW超声的手机无接触手势识别
**目标**: 利用17-23kHz FMCW超声信号，通过微多普勒特征+CNN实现6种空中手势识别
**平台**: Android采集端 (AAudio低延迟管线) + PC Python处理端
**手势集**: Push(前推), Pull(后拉), Sweep(横扫), Slide(滑动), Fist_bump(碰拳), Grab(抓取)

## 2. 信号处理管线

### 2.1 FMCW参数设计 (2026-05)

| 参数 | 值 | 依据 |
|------|-----|------|
| 采样率 | 48000 Hz | 手机麦克风标准采样率 |
| Chirp频率范围 | 17000-23000 Hz | 超声波段，人耳不可闻，手机扬声器可发射 |
| 带宽 | 6000 Hz | 距离分辨率 ≈ 343/(2×6000) ≈ 2.86cm |
| Chirp周期 | 20ms (960采样) | 短周期保证慢时间采样率 |
| 占空比 | 100% | 连续发射，最大信号能量 |
| 慢时间采样率 | 50 Hz (Nyquist=25Hz) | stride=960，无帧重叠 |
| 距离门范围 | 0-100cm | 实际手势距离10-30cm |

### 2.2 信号处理流程 (V3/V4)

```
PCM(float32) → 6阶Butter带通滤波(17-23kHz) → 互相关对齐 → 滑动窗口提取chirp
→ 静态背景去除(中值) → 去斜(dechirp) → 6阶低通滤波(2.5kHz)
→ 距离FFT → 最强距离门选择 → 相位差分 → 微多普勒
→ STFT → per-frequency median subtraction(去除硬件固定频率干扰)
```

关键设计决策:
- **无帧重叠**: 试过50%重叠(100Hz慢时间)→Doppler谱模糊，回退无重叠50Hz
- **静态背景去除**: 中值滤波比均值对异常帧更鲁棒
- **最强距离门**: 选能量最大的距离bin，实际大部分样本距离在0-3cm（直达波为主）
- **per-frequency median subtraction** (2026-05-27): 新手机在-0.5Hz和-1.1Hz出现固定横线，STFT后对每频率bin减中值消除

### 2.3 特征提取

**幅度包络**: 最强距离门复信号取模 → Savitzky-Golay平滑(25帧=0.5s窗, 3阶多项式)
**微多普勒谱图**: 最强距离门±5bin平均 → STFT(32点Hann, 28重叠) → abs → gamma=0.5幂压缩 → Gaussian平滑(sigma=0.6)

幂压缩参数选择: 弱信号（手势回波）比直达波低20-30dB，gamma=0.5有效提亮弱信号，过高会放大噪声。
Gaussian sigma=0.6: 轻微平滑，保留时间方向细节。

### 2.4 放弃方案

| 方案 | 问题 | 时间 |
|------|------|------|
| 60%占空比(方案A) | 信号断续，慢时间噪声增大 | 2026-05 |
| 50%重叠帧提取(V4) | 慢时间100Hz但多普勒谱质量反而下降 | 2026-05 |
| STFT相位差分谱(图6) | 相位噪声太大，完全不可用 | 2026-05 |
| 自适应多普勒Y轴范围 | 用户要求固定范围以便对比 | 2026-05 |
| 画圈/旋转手势 | 切向运动径向分量弱，微多普勒不可见 | 2026-05 |
| 多进程加速(Windows spawn) | 进程启动+导入开销>加速收益 | 2026-05 |
| 多线程加速 | matplotlib GIL争用，反而不如单线程 | 2026-05 |
| plt.style.use("fast") | 效果不明显，已回退 | 2026-05 |

## 4. CNN模型与训练

### 4.1 模型架构

**双流CNN** (DualStreamCNN):
- **谱图分支**: Conv2d(32×26) → 3层卷积+池化 → FC(128)
- **包络分支**: Conv1d(100) → 3层卷积+池化 → FC(128)
- **融合**: 拼接(256) → FC(64) → Dropout(0.3) → FC(N类)
- **总参数量**: ~1.1M

设计理由:
- 谱图用2D卷积捕获时频二维模式，包络用1D卷积捕获时序变化
- 双流分离避免一种模态主导另一种
- Dropout(0.5)在两分支输出，Dropout(0.3)在分类头

### 4.2 训练配置

| 参数 | 值 |
|------|-----|
| 框架 | PyTorch |
| 损失函数 | CrossEntropyLoss |
| 优化器 | Adam (lr=0.001, weight_decay=1e-4) |
| 学习率调度 | CosineAnnealingLR (T_max=80) |
| Batch size | 8 |
| Epochs | 80 |
| 验证策略 | 5折分层交叉验证 |

### 4.3 数据增强

- 高斯噪声 (σ=0.02, 概率50%)
- 时间平移 (±3帧, 概率50%)
- 幅度缩放 (0.85~1.15, 概率30%)

增强设计理由: 手势动作时间位置和速度有自然变化，噪声增强模拟不同信噪比。

### 4.4 训练结果 (2026-05-24)

**Push vs Pull vs Sweep 三分类** (150样本, 每类50):

| 指标 | 值 |
|------|-----|
| 5-Fold CV准确率 | 100.00% |
| Push Precision/Recall | 1.000/1.000 |
| Pull Precision/Recall | 1.000/1.000 |
| Sweep Precision/Recall | 1.000/1.000 |

### 4.5 标签打乱验证 (2026-05-24)

**问题**: 三分类100%准确率是否真实？是否存在数据泄漏？

**方法**: 将样本标签随机打乱，重新5折CV训练。若准确率仍高→存在泄露；若降至随机水平(~33%)→真实学习。

**结果**:
| 验证方式 | 准确率 |
|----------|--------|
| 正确标签 | 100.00% (±0.00%) |
| 随机打乱标签 | 26.00% (±6.46%) |

**结论**: 打乱后接近随机水平(33.3%)，证明模型学到的是真实特征差异。100%准确率是因为Push/Pull/Sweep的多普勒模式差异极大:
- Push: 正多普勒（靠近）
- Pull: 负多普勒（远离）
- Sweep: 振荡多普勒（左右来回）

预期当加入Fist_bump、Grab等模式相似的手势后，准确率会下降，真正的分类挑战才出现。

### 5.3 C++ 信号处理管线移植 (2026-05-24 → 2026-05-25)

**目标**: 将 Python `_process_core()` 完整移植到 Android C++，与 Python 逐步骤对齐。

**文件结构**:
- `fft.h`: 轻量 radix-2 FFT(R2FFT, 32点) + Bluestein任意长度FFT(960点)，预计算旋转因子+bit-reversal表
- `biquad.h`: Direct Form I 级联滤波器，模板化支持N阶，预定义 BandpassFilter/LowpassFilter 别名，含 `processFiltFilt()` 双向零相位滤波
- `dsp.h`: **ChirpProcessor** 类，12步完整管线(header-only):
  1. 带通滤波(filtfilt) 2. 互相关对齐 3. chirp帧提取 4. 中值背景去除
  5. 去斜 6. 低通滤波(filtfilt, 实部虚部分离) 7. 距离FFT(960点Bluestein) 8. 最强距离门
  9. 相位差分微多普勒 10. SG平滑包络 11. STFT微多普勒谱图
  12. 幂压缩+高斯平滑(无resize，保持原始尺寸)
- `biquad_coeffs.h` / `constants.h`: gen_constants.py生成，全部预计算常量

**2026-05-25 向Python对齐的5处修正**:

| 步骤 | 修改前 (不对齐) | 修改后 (对齐Python) | 影响 |
|------|----------------|-------------------|------|
| 带通滤波 | forward-only | filtfilt双向零相位 | 频率响应一致 |
| 低通滤波 | 单实例先后处理实虚部(串扰) | 独立两实例分离处理+filtfilt | 消除状态泄漏 |
| 距离FFT | 1024点零填充 | 960点Bluestein | bin间距50Hz/bin一致 |
| 距离门 | 全部480bin | kValidRangeBins=35 (≤1m) | 与Python r_mask一致 |
| 输出resize | 双线性插值到32×26+100 | 不做resize | 避免时间轴拉伸纹理变糊 |

**STFT时间覆盖**:
- STFT窗(32帧=640ms)占用前640ms，输出覆盖 0.32s~1.60s，共约17列(取决于nChirps)
- 谱图行=32频点，时间步间距=80ms(4帧hop)
- CNN推理时 resize 到 32×26 输入尺寸，不影响训练-推理一致性

**识别架构**:
- Batch模式: 收集完整2s录音 → 一次性处理 → CNN推理
- 不能流式原因: 中值背景去除需要全局所有chirp、STFT覆盖整个时间窗
- 处理+推理总延迟约200ms（2s录音后），交互体验可接受

**双模式设计**:
- 录制模式: 照旧，录音→存PCM文件
- 识别模式: 录音2s→ChirpProcessor→谱图+包络→JNI回传Java渲染
- 两模式互斥，录音回调根据 `isRecognition` 标志分流数据

**UI变更**:
- 布局分录制/识别两个互斥区域，模式切换按钮在底部
- 识别区域含谱图渲染（Jet colormap, 256×208像素位图）
- 双Space weight布局实现内容居中+按钮底部对齐

**ONNX Runtime CNN推理** (2026-05-24):
- LibTorch Android AAR 无法下载，改用 ONNX Runtime C API
- `export_onnx.py`: 将 DualStreamCNN 导出为 `model.onnx` (opset 17)
- `native-lib.cpp`: 加载 assets/model.onnx + 推理 predictGesture()
- 推理流程: 谱图(32×26)+包络(100) → min-max归一化 → CreateTensor → Run() → softmax → JSON返回
- 分类结果: {"class":0/1/2, "name":"Push"/"Pull"/"Sweep", "probs":[...]}
- 当前仅训练了 Push/Pull/Sweep 三分类 (150样本, 5-Fold CV 100%)
- 模型扩展: 采集更多手势后重新训练 → 导出新 onnx → 替换 assets/model.onnx

### 5.5 ONNX Runtime NEON SIMD Gemm Bug (2026-05-26)

**现象**: 四/五分类模型部署后，最后一个或倒数第二个输出通道恒为精确0.00000000。

**定位**:
- Python ORT 1.26.0全类正常，Android ORT 1.21.0某通道恒0
- 排除: 模型权重(bias≠0)、特征全零(spec sum正常)、数据泄漏(标签打乱21%)
- 三分类正常(3输出无zero)，四/五分类有zero
- 切换优化级别仅改变zero在哪个通道，不修复
